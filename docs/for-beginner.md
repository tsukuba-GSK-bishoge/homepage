## Markdown

> Markdownとはプレーンテキスト形式で書式付きテキストを記述する軽量マークアップ言語です。

[Wikipedia](https://ja.wikipedia.org/wiki/Markdown)

プレーンテキストというのは純粋なテキストを指します。メモ帳で開いて中身が読めるものがプレーンテキストと言っていいと思います。
これの対になる概念が**バイナリ**です。これは2進数で書かれており、基本的にはコンピューターが読むものだと思えば大丈夫です。例としては`.exe`ファイルや`.pdf`ファイルや`.docx`ファイルなどが該当します。これらは人間が中身を直接見ても理解できず、コンピューターが解釈することで初めて画像や動画、実行ファイルとしての意味を持ちます。

書式付きテキストというのは見出しとか引用とか強調がついているテキストのことです。Wordで書いた文章をイメージしてもらうといいかもしれません。

マークアップ言語は、見出しとかの文章の構造を記述する言語だと思っていただければ大丈夫です。一応例を上げるとHTMLとかLaTeXなどがあります。

さて、ここまでを読むとWordでいいじゃんとなってしまうかもしれません。しかしMarkdownとWordには大きな差があります。

### 特定のソフトウェアに依存していない

Wordは当然ながらWordのソフトウェアを使わないと使うことができません。
しかしMarkdownはプレーンテキストなのでメモ帳で読み書きすることができます。
この点でWordに比べて自由であると言えます。

### 簡単かつ素早く書ける

Wordを年単位で使っていないので変わっていたら申し訳ないのですが、Wordでは見出しや書式などを画面上のメニューから設定することが多いです。
一方でMarkdownで見出しを書きたいだけなら`#` `<Space>`のあとに見出しを書くだけで終わります。
このように素早く、簡単に文字を記述することができます。

### きれいに表示される

Wordを使っているとよくわからない隙間が発生するとか、数式がきれいに書けないとか、いろいろ不便な点が出てきます。
Markdownでは基本的にはコンピューター上で文書を見る分には十分きれいな書式で表示してくれます。
印刷したければpdfに変換して印刷するとかもできます。

### 情報を容易に出し入れできる

変な表現ですが簡単にいうとコピペしやすい、ということです。
Wordでは他所でコピーした文をそのままペーストすると書式が崩れたり、謎の問題が起きたりします。
またMarkdownではもとのソースをコピーしやすいということがあります。
具体的にはこの文章でわからない部分があるな、という場合にはソースを直接開いてコピーすれば簡単に調べることができます。
なんならこの.mdファイルを直接AIに読み込ませて逐次解説をさせる、ということができます。

ここまで読んできたあなたは早くWordを捨ててMarkdownを利用したいという気持ちになってきました。
最初は書き方がわからないことも多いと思います。
Markdown記法は様々な書式があって全部覚える必要はありません。
これどうやって表現すればいいんだろう、と思ったら都度検索していくとよく使う書き方は覚えると思います。

一応参考としてチートシートを紹介しておきます。

[markdownチートシート](https://www.tohoho-web.com/ex/markdown.html)

## ターミナル

ターミナルは文字列でPCを操作するツールです。
イメージとしてはハッカーが操作しているあの黒い画面です。

ターミナルに限らない話ですがエラー文は怖いものではありません。
僕たちのミスをコンピューター側が教えてくれているものと考えておきましょう。

ここではターミナルとしてPowerShellを例に説明します。
Windowsのアプリを検索する場所からPowerShellと入力して起動しましょう。
VSCode内でターミナルを開くことができますがここでもPowerShellを利用することができます。

### コマンド

ターミナルではコマンドを打って操作します。ターミナル上では今どのフォルダにいるかが表示されます。

例)

```
PS C:\Users\Hayato\Downloads>
```

この例では`C:\Users\Hayato\Downloads`にいます。エクスプローラーでダウンロードと表示される場所と同じです。

#### cd

`cd`はフォルダを移動するコマンドです。`cd folder`のように指定することで移動することができます。

では先ほどの場所から`cd`を使って`src`に移動してみましょう

例)

```
PS C:\Users\Hayato\homepage> cd src
PS C:\Users\Hayato\homepage\src>
```

何も表示されませんでしたが、表示されているフォルダを見ると移動できている事がわかります。(このようにコマンド上で何も表示されないという結果は基本的にはコマンドの成功を意味しています)

また上のフォルダに戻るには`cd ..`で移動します。現在のフォルダを`.` 親のフォルダを`..`で表現します。

例)

```
PS C:\Users\Hayato\homepage\src> cd ..
PS C:\Users\Hayato\homepage>
```

成功しましたね。

またこのように親子関係のフォルダに移動するだけでなく、パス(フォルダの場所)を直接指定することで移動する事もできます。

例)

```
PS C:\Users\Hayato\homepage> cd C:\Users\Hayato\homepage\src\assets\
PS C:\Users\Hayato\homepage\src\assets>
```

このようにパスを直接指定することでも移動できます。

## Webサイトの仕組み

ここではWebの仕組みを大雑把に説明します。
例としてこの美少女ゲーム班のWebサイトを見るときのことを考えてみましょう。
ではまずPCのブラウザから美少女ゲーム班のWebサイトを開いた瞬間について考えます。

手元のPCからインターネットを通じて美少女ゲーム班のサーバーに美少女ゲーム班のサイトの情報をくれ、というリクエストが飛びます。
(ここではサーバーについて詳しく触れません。巨大な一つのPCだと思ってください)
これをHTTPリクエストといいます。
ここで美少女ゲーム班のサーバーは**ステータスコード**と**ヘッダー**と**ボディ**をレスポンスとして返します。

- ステータスコード:リクエストの結果を示す番号です(例:200は成功,404はページが見つからない等)
- ヘッダー:サーバーやレスポンスの追加情報を提供する(あんまり気にしなくてok)
- ボディ:レスポンスの中身です。今回の例ではWebサイトの中身が`HTML`で返ってきます。

ここでターミナルからHTTPリクエストを送ってみましょう。

```
PS C:\Users\hayato> curl.exe -i https://bishojo.gsk-tsukuba.net
HTTP/1.1 200 OK
Accept-Ranges: bytes
Content-Length: 15127
Content-Type: text/html
Date: Sun, 19 Apr 2026 09:15:29 GMT
Etag: "69e0fe56-3b17"
Last-Modified: Thu, 16 Apr 2026 15:20:54 GMT
Server: nginx/1.29.8

<!DOCTYPE html>
<html lang="ja">
  <head>
    <!-- meta settings start -->
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <meta name="theme-color" content="#ff8bd3">
    <meta name="generator" content="Astro v5.18.1">
    <meta
      name="description"
      content="こちらは筑波大学公認サークルである現代視覚文化研究会 美少女ゲーム制作班の公式サイトです。"
    >
  </head>
  <body>
    ...
  </body>
</html>
...
(省略)

```

ステータスコードを見ると`HTTP/1.1 200 OK`というステータスコードが帰ってきている事がわかります。

またその後にヘッダーがあり、Webサイトの情報が返ってきています。

その後の`<!DOCTYPE>`という部分から文章が始まっており、ここでWebサイトの構造が記述されています。

またWebサイトは`HTML`だけで記述されているわけではなく、`CSS`と`JavaScript`によって構成されます。
`CSS`はWebサイト上の文字の大きさや、色合いなどの見た目の情報を記述しており、`JavaScript`はボタンを押したときなどのWebサイトの動きを記述しています。
先程はトップページの`HTML`を取得したので、このレスポンスでは`HTML`が返ってきています。実際のブラウザ上では、この`HTML`を読んだあとで必要な`CSS`や`JavaScript`、画像などを自動で追加でリクエストし、Webサイトの見た目を構築しています。

## Node.js

Node.jsは`JavaScript`を手元のPC上で動作させるための実行環境です。
`JavaScript`はWebサイトの動きを記述するための言語なのでブラウザ上でしか動作できません。
これを手元のPC上で動かすための実行環境なわけです。

`JavaScript`はブラウザという試験管の中でしか生きられない生き物なのでそれを試験管の外でも生きられる環境を作ってやるソフトウェアというわけです。

### npmについて

Node.jsを利用する際には`npm`というものも同時に利用すると思います。

これはNode.jsのパッケージ管理ツールです。
パッケージというのは`React`や`jQuery`などが例にあります。
パッケージは誰かが作ったコードなどを、まとめて利用できるようにしたものです。

日付を管理しやすくする機能や、UIを使う機能がパッケージとして公開されています。

例に挙げると`jQuery`はWebページの見た目や動きを`JavaScript`で簡単に操作するためのパッケージです。
(Tyrano Scriptではこれが使われています。)

### インストール方法

[Node.js](https://nodejs.org/ja/download/current?utm_source=chatgpt.com)

上のサイトからWindowsインストーラー(.msi)と書かれたボタンをクリックし、ダウンロードした.msiファイルを実行します。

ターミナル上で

```
node -v
npm -v
```

と入力してバージョンが表示されたら成功です。

美少女ゲーム班のWebサイトではNode.jsを利用しているので、Node.jsをインストールすると変更した内容などを手元で実際にWebサイトを起動して確認できるようになります。

## Astro

AstroはWebページを構築するためのパッケージです。

プロジェクト内にある情報を組み立ててWebサイトを作ってくれるパッケージです。

具体的にはこのリポジトリの`src/contents/works`を見ると`Markdown`ファイルが置かれていると思います。
Astroでは`Markdown`ファイルを`HTML`に変換し、Webページとして表示してくれます。
このような細かな素材(コンポーネントといいます)を組み合わせて実際のWebサイトを構築するのがAstroの機能になっています。

Astroは高速で、必要のない`JavaScript`をできるだけ減らしやすいという特徴を持っています。

このような特徴から、閲覧がメインになるWebサイトの開発が得意です。
ただしログイン機能が必要なサイトなどの複雑なアプリでは、他の選択肢も含めて検討されることが多いです。

美少女ゲーム班のサイトでは班の紹介などがメインなのでAstroが使われているわけですね。

## GitおよびGitHub

### gitとは?

gitはファイルの変更履歴を記録するためのツールです。
雑にいうと「セーブポイントを作れるシステム」です。

例えばWebサイトを編集していて、変な変更をして壊してしまったとします。
そのときgitで途中の状態を記録しておけば、あとから「昨日の正常な状態」に戻したり、「どのファイルをどう変えたか」を確認したりできます。

またgitは一人で使っても便利ですが、複数人で開発するときに特に強力です。
誰がどの変更をしたかを追いやすく、別々に作業した内容をあとからまとめることもできます。

### GitHubとは?

GitHubはgitで管理しているデータをインターネット上に置いておくサービスです。
gitそのものは手元のPCだけでも使えますが、GitHubに置くことで複数人で同じリポジトリを共有しやすくなります。

このリポジトリもGitHub上に置かれており、そこからダウンロードしたり、変更をアップロードしたりできます。

要するに

- `git`: 変更履歴を管理する仕組み
- `GitHub`: gitのデータを置いたり共同開発したりする場所

という理解で最初は十分です。

### gitの構造

gitでは作業しているファイルがいくつかの段階に分かれています。
最初はこの段階の違いがわかりにくいのですが、ここがわかるとかなり理解しやすくなります。

#### working tree

今まさに自分が編集しているファイル群です。
VSCodeで書き換えたけれどまだgitに記録していない状態はここにあります。

#### staging area

次の記録に入れたい変更を一時的に置いておく場所です。
「この変更は今回の記録に含める」「これはまだ含めない」という選別をするためにあります。

#### commit

変更を記録したセーブポイントです。
`commit`を作ると、その時点での変更が履歴として残ります。

流れとしては

`working tree` -> `staging area` -> `commit`

という順番になります。

ゲームのセーブに例えると、ファイルを編集している状態が`working tree`、セーブする内容を確認している状態が`staging area`、実際にセーブしたものが`commit`です。

#### branch

branchは変更の流れが枝分かれしたものです。
例えば本番用の安定した流れとは別に、新しい機能を試すための流れを作ることができます。

このリポジトリでは普段は`master`ブランチが中心になっています。
ただし直接`master`をいじるのではなく、作業用のbranchを切ってそこから変更を送ることが多いです。
初心者向けに強く言うと、`main`や`master`のような本体のbranchに直接pushするのはやめましょう。
必ず作業用のbranchで変更して、Pull Requestを作ってから取り込んでもらうほうが安全です。

### gitの使い方

ここでは最低限よく使う流れだけ説明します。

#### clone

GitHub上にあるリポジトリを自分のPCにコピーする操作です。

```
git clone https://github.com/tsukuba-GSK-bishoge/homepage.git
```

これでこのリポジトリ一式を手元に持ってこれます。

#### status

今の状態を確認するコマンドです。
困ったらとりあえずこれを打つとよいです。

```
git status
```

どのファイルが変更されたか、`add`されているか、今どのbranchにいるかなどが確認できます。

#### add

変更したファイルをstaging areaに入れるコマンドです。

```
git add docs/for-beginner.md
```

全部まとめて入れたければ

```
git add .
```

でも大丈夫です。ただし意図しないファイルまで含めやすいので、慣れないうちはファイルを指定したほうが安全です。

#### commit

staging areaに入れた変更を記録します。

```
git commit -m "docs: add git explanation for beginners"
```

`-m`の後ろにはこの変更が何なのかを短く書きます。
後から見返したときにわかる文章にするとよいです。

#### pull

GitHub上の最新の変更を自分のPCに取り込むコマンドです。
他の人が更新しているリポジトリではよく使います。

```
git pull
```

#### push

自分が作ったcommitをGitHubにアップロードするコマンドです。

```
git push
```

ただしその前に、いきなりpushしないで`npm run build`が通るか確認してください。
buildが失敗する状態でpushすると、サイトが壊れたり、あとで直す手間が増えたりします。
**また`main`や`master`に直接pushするのではなく、作業用のbranchにpushしてPull Requestを作るようにしてください。**

この流れをまとめると

1. ファイルを編集する
2. `git status`で状態を見る
3. `git add`で今回記録したい変更を選ぶ
4. `git commit`で記録する
5. `npm run build`が通るか確認する
6. `git push`で作業用branchをGitHubに送る

となります。

#### diff

何を変更したかを見るコマンドです。

```
git diff
```

commit前に「本当にこの変更で合っているか」を確認するのに便利です。

### 共同開発のススメ

一人で編集していると、どこで壊れたのかわからなくなったり、変更が大きくなりすぎたりします。
gitとGitHubを使うなら、なるべく小さく区切って記録するのがおすすめです。

例えば

- 文章を1本追加した
- 画像を差し替えた
- 見出しのデザインを調整した

など、意味のまとまりごとにcommitすると後から見返しやすくなります。

また複数人で作業するときは、自分専用のbranchを作ってそこで編集し、終わったらPull Requestを作るのがよくある流れです。

Pull Requestは「この変更を本体に取り込んでください」とお願いする仕組みです。
差分を見ながらレビューしてもらえるので、事故を減らしやすいです。

最初は難しく感じると思いますが、壊すことを恐れすぎなくて大丈夫です。
gitがあると戻りやすいので、むしろ色々試しやすくなります。

### CI/CDとは?

CI/CDは開発を自動化する仕組みのことです。

#### CI

CIは`Continuous Integration`の略です。
日本語だと継続的インテグレーションなどと言います。

簡単にいうと、コードをGitHubに送ったときに自動でチェックしてくれる仕組みです。
このリポジトリではPull Requestを出したときに、型チェックやESLintやPrettierの確認が自動で走ります。

人間が毎回手で確認しなくても、機械がまずミスを見つけてくれるわけですね。

#### CD

CDは`Continuous Deployment`の略です。
こちらは変更を本番環境へ自動で反映する仕組みです。

このリポジトリでは`master`ブランチにマージされると、自動でデプロイされてWebサイトに反映されます。

つまり

- `CI`: 壊れていないか自動で検査する
- `CD`: 通った変更を自動で公開する

という役割です。

初心者のうちは細かな仕組みまで理解しなくても大丈夫です。
「GitHubに変更を送ると自動で確認してくれる仕組みがある」くらいの理解がまずあれば十分です。
